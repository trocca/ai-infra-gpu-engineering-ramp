Killer facts — the numbers & names to know cold

KAI Scheduler

Open-sourced Run:ai scheduler. Apache 2.0, github.com/NVIDIA/KAI-Scheduler. Launched KubeCon EU, Apr 1 2025. Run:ai acquired ~$700M, closed Dec 30 2024. Runs alongside kube-scheduler.

gang schedulingDRF fair-sharefractional GPU

Dynamic Resource Allocation

Replaced the device-plugin integer count model. alpha v1.26 → beta v1.32 → core GA v1.34 (resource.k8s.io/v1). Four objects: ResourceClaim, ResourceClaimTemplate, DeviceClass, ResourceSlice.

say "GA v1.34"

ComputeDomains (GB200)

GB200 NVL72 = 72 GPUs / 18 nodes / 9 NVLink switches, ~1.8 TB/s per-GPU. IMEX = the driver memory-exchange service; a ComputeDomain CRD provisions one isolated IMEX domain per workload. Placement via nvidia.com/gpu.clique.

NVIDIA Dynamo

GTC Mar 2025, 1.0 GA Mar 2026. Disaggregated prefill/decode. 4 parts: Smart Router (KV-aware), Planner, KVBM (HBM→DRAM→SSD→object), NIXL. Engine-agnostic. 30× on DeepSeek-R1 / GB200.

successor to Triton

Workload APIs

Kubeflow Trainer / TrainJob v2 — GA Jul 21 2025, one unified CRD on JobSet. LWS — leader+workers as one unit (multi-node vLLM). Grove — multi-component disaggregated inference, open-sourced Nov 10 2025.

PyTorch parallelism

DDP (fits 1 GPU) → FSDP (ZeRO-3) → FSDP2 (fully_shard, DTensor, composes w/ TP) → Megatron-Core TP×PP×CP×EP×DP (5D) → NeMo 2.0.

NCCL

Collectives (AllReduce…). Ring vs Tree vs NVLS. Transports auto-picked: NVLink intra-node → IB/RoCE + GPUDirect RDMA inter-node. SHARP = in-network reduction (2.27, ≤6 SMs). nvidia_peermem enables GDR.

NGC container

Tag YY.MM-pyN = calendar build (e.g. 26.04-py3 = Apr 2026). Monthly ABI-matched stack. libcuda (driver) vs libcudart (toolkit/container). CUDA 13.x needs driver ≥580.

GPU sharing — 3 ways

MIG = HW partition, hard mem+fault isolation. Time-slicing = SW round-robin, no isolation. MPS = concurrent contexts, weak isolation. KAI fractional on top (no mem isolation by default → +MIG/HAMi).

The abstraction pitch — your "teach us something" talk

"Training frontier models used to need a team of distributed-systems PhDs. Today a developer writes train.py, a Dockerfile, and ~30 lines of YAML — everything underneath is automatic. That line, between intent and mechanism, is the product."

1 · "Here's all you write" (L15,10,9)

Three code blocks on one slide: a 20-line train.py, a 4-line Dockerfile, a ~30-line TrainJob. "Nothing below this slide is written by the ML engineer."

2 · The operator makes distributed "just work" (L8)

Headless Service → stable pod DNS. Operator injects MASTER_ADDR/RANK/WORLD_SIZE. torchrun reads them, spawns one proc/GPU. Dev wrote none of it.

3 · The scheduler wins the GPU dollars (L7–5)

Gang scheduling (draw the deadlock!) + fractional sharing + topology placement + DRA ComputeDomains. "A cluster at 30% is a cluster you're 3× overpaying for."

4 · The node just becomes a GPU node (L4–0)

One Helm chart (GPU Operator) → driver, toolkit, DCGM, MIG. CDI injects /dev/nvidia*. "You never SSH into a node."

Three whiteboard set-pieces (draw from memory)

  • Partial-gang deadlock: Job A grabs 6/8 GPUs & waits, Job B grabs 2 & waits → deadlock. Gang scheduling = all-or-nothing → forward progress.
  • Disaggregated serving: prefill (compute-bound, low TP) vs decode (bandwidth-bound, high TP) → split into independent pools → 30× throughput.
  • Compatibility line: libcuda (driver, bottom) vs libcudart (container, top); forward-compat lets new CUDA run on old driver.

Always translate tech → money

Gang scheduling → no deadlocked GPU-hours. Fractional+fair-share → 30%→70% utilization = 2× fewer GPUs. Topology → faster time-to-model. Abstraction → fewer specialist hires. Open-source (KAI/Dynamo/DRA) → no lock-in. AI Enterprise → supported path to prod.

Pre-sales objection handling

ObjectionCrisp answer
"Our GPUs sit at 25%."KAI fair-share queues + fractional GPU + bin-packing + preemption; separate training/inference pools; MIG for guaranteed inference QoS.
"How do we share GPUs safely?"MIG (HW, hard isolation) vs time-slicing (no isolation) vs MPS (weak); + KAI fractional & quotas. Know the trade-offs.
"What's the ROI?"AI-factory math: tokens/sec/$ & utilization; software sub vs hardware TCO; DGX Cloud for burst.
"1000-GPU jobs fail — then what?"Gang scheduling avoids partial placement; NeMo checkpoint/restart; torchrun elasticity; DCGM health-evict; auto-requeue.
"Observability?"DCGM + DCGM-Exporter → Prometheus/Grafana; per-workload GPU metrics; Run:ai dashboards; NVLink telemetry.
"Vendor lock-in?"KAI (Apache 2.0), Dynamo (open), DRA/CDI/LWS upstream, Nemotron open models. AI Enterprise is the supported option, not the only one.
"Networking bottleneck at scale?"Spectrum-X / Quantum IB + GPUDirect RDMA + rail-optimized topology + SHARP; scheduler topology-awareness cuts cross-rack traffic.

Correct these misconceptions cleanly

  • KAI runs alongside kube-scheduler, doesn't replace it.
  • DRA core is GA (v1.34), not experimental.
  • Time-slicing does not isolate memory — only MIG does.
  • ComputeDomains ≠ NVLink; they isolate per-workload IMEX domains on top of it.
  • GPUDirect RDMA (NIC↔GPU, inter-node) ≠ NVLink (GPU↔GPU, intra-rack).
  • Don't run device plugin + DRA GPU driver on the same GPUs.
